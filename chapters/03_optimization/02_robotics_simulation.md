# 第三章 優化 Optimization

## 2. 進入模擬環境 Robotics Simulation

在機器人軟硬體整合的生命週期中，「實體測試」往往伴隨著極高的物理風險（如機器人失控撞牆、機械臂碰撞受損）與高昂的人力與時間成本。
因此，在物理實體硬體尚未完全組裝或量產完成前，建構一個高逼真度的物理模擬環境，是現代機器人研發流程中不可或缺的黃金策略。

---

### 2.1 模擬環境的商業與研發價值

在實務工程上，引入模擬環境能帶來三大顯著優勢：

1.  **極大降低研發與測試成本**：
    *   **費用成本**：一台工業級 3D 光達或人型機器人本體動輒數十萬至數百萬台幣。在模擬器中，開發者可以「免費」掛載任意數量的虛擬光達、高解析度相機，並在虛擬空間中部署整隻機器人車隊，無需等待硬體採購。
    *   **人力與時間**：實體測試需要人員將機器人搬運至特定場地、佈置環境（如擺放標定板、障礙物）、充電、並手動記錄數據。模擬環境支援自動化、並行化（Parallelized）測試，能 24 小時不間斷地執行演算法回歸測試（Regression Testing）。
2.  **安全且可重複的「邊緣極端案例」測試 (Edge Cases)**：
    *   在現實世界中，測試機器人在「暴雨」、「濃霧」、「強光直射相機」或「光滑鏡面反射」等極端天氣與環境下的防撞避障性能是非常困難且危險的。模擬器中可以精準控制物理與氣象參數，讓演算法在安全的虛擬空間中不斷試錯與優化。
3.  **無縫銜接 AI 數據訓練 (Synthetic Data)**：
    *   深度學習演算法（如 YOLO 或語意分割）需要海量的標註數據。模擬器支援一鍵輸出自動標註好的深度圖、語意分割圖與 3D 邊界框（Bounding Boxes），徹底解決人工標註資料效率低下的痛點。

---

### 2.2 主流模擬平台：Gazebo vs. NVIDIA Isaac Sim

目前 ROS 2 生態系中最主流的兩大模擬平台各自扮演著不同的角色：

| 評估維度 | Gazebo (Classic / Ignition) | NVIDIA Isaac Sim |
| :--- | :--- | :--- |
| **整合難易度** | **極高**。ROS 2 官方原生支援，社群資源極其豐富。 | **中等**。需透過 Omniverse Bridge 或 ROS 2 Extension 串接。 |
| **渲染與畫質** | 偏向基礎幾何渲染，畫面擬真度一般，適合傳統 2D/3D SLAM 與控制驗證。 | **頂級**。基於 NVIDIA RTX 光線追蹤（Ray Tracing），擁有電影級渲染品質，能完美模擬真實世界光影。 |
| **感測器物理模擬** | 基礎感測器噪訊模擬，對環境材質反射（反射率、吸收率）物理擬真較弱。 | **極強**。能高擬真地模擬紅外線相機光照、Lidar 雷射脈衝物理反射、超音波回波等物理特性。 |
| **運算平台要求** | 輕量，一般 CPU 搭配基礎獨立顯示卡即可流暢運行。 | 沉重。必須配備 NVIDIA RTX 顯示卡，強烈依賴 GPU 進行物理與光線追蹤計算。 |
| **適用場景** | 傳統移動機器人（AGV）導航、機械臂路徑規劃。 | 深度學習感知訓練、極端環境測試、強化學習（Reinforcement Learning）與人型機器人步態訓練。 |

### 2.3 核心技術議題：3D 機器人模型格式轉換

不論選擇哪一個模擬平台，開發者面臨的第一個共同技術門檻，就是如何將機構工程師繪製的工業 CAD 模型轉化為模擬器看得懂的格式：

```
工業 CAD 軟體 (SolidWorks / Creo / Inventor) 
       │
       ▼ [導出/轉換]
CAD / STEP 標準幾何模型
       │
       ├─► 轉為 URDF / SDF (適用於 Gazebo) ──► 結合 Mesh 簡化 (減少 STL 面片數) 與慣性張量計算
       │
       └─► 轉為 USD (適用於 Isaac Sim) ──► 保留豐富材質、貼圖與光學物理屬性
```

#### 2.3.1 模型簡化與物理參數的重要性
*   **網格簡化 (Mesh Simplification)**：
    機械工程師導出的 3D 模型通常極為精細（包含所有螺絲、內部齒輪）。如果直接將高精度的 STL 網格導入模擬器，會瞬間耗盡顯示卡記憶體，導致模擬 FPS 暴跌。因此，必須先進行網格簡化，用簡單的幾何體（凸包, Convex Hull）來代替機器人的物理碰撞體（Collision Model），僅保留精簡後的網格作為外觀顯示模型（Visual Model）。
*   **慣性矩張量 (Inertia Tensor) 與質量**：
    在 URDF/SDF 中，每個剛體（Link）都必須被正確填入其物理質量（Mass）與慣性張量（Inertia $I_{xx}, I_{yy}, I_{zz}$）。若這些數值是亂填的（例如填 `0` 或是單位不對），模擬器中的物理引擎（如 ODE, PhysX）在計算重力與接觸力時，就會導致機器人在虛擬世界中異常劇烈發抖、甚至直接飛出地圖。
