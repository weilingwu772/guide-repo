# 第三章 優化 Optimization

## 1. 淺談校正實務 Is the Data Really Accurate?

在完成了感測器的對接與前處理後，開發者往往會面臨一個非常現實的工程挑戰：**「感測器源源不絕送出來的數據，真的是準確且及時的嗎？」**。本節將探討感知實務上的資料品質診斷、異常處理，以及移動系統的精準校正。

---

### 1.1 實務面診斷：我的相機真的有 30 FPS 嗎？

在軟體設定中，我們常將相機或雷達的採樣率（Frame Rate / Frequency）設定為例如 30 Hz（30 FPS）。但在複雜的嵌入式運算、高 CPU 負載或 USB 頻寬競爭下，數據往往會發生**「掉幀」**、**「延遲 (Latency)」** 甚至 **「阻塞 (Blocking)」**。

#### 1.1.1 如何使用 CLI 工具進行診斷
在 ROS 2 中，我們可以利用自省工具（Introspection Tools）快速評估數據流品質：

1.  **檢查實際發布頻率**：
    ```bash
    ros2 topic hz /camera/color/image_raw
    ```
    *   **診斷分析**：若輸出結果波動劇烈，或平均頻率遠低於設定值（例如僅有 20 Hz），則說明系統發生了**掉幀**。
2.  **檢查資料傳輸延遲**：
    ```bash
    ros2 topic delay /camera/color/image_raw
    ```
    *   **診斷分析**：此指令會比對 Message Header 中的時間戳（由驅動節點在硬體採樣時打上）與當前系統時間的差值。若延遲大於一幀的間隔（例如 30 Hz 下大於 33 ms，甚至達到 200 ms），說明數據在硬體驅動、USB 傳輸或記憶體拷貝中發生了嚴重的**延遲**與**積壓**。

#### 1.1.2 資料異常（掉幀、延遲、阻塞）的優化處理方案

當發現數據流不穩定時，開發團隊可以從以下三個維度進行工程調優：

```
硬體與頻寬層級 (Host & Bandwidth) ──> DDS 服務品質 (QoS Policy) ──> 驅動架構設計 (Driver Architecture)
```

1.  **硬體與底層通道優化**：
    *   **USB 頻寬競爭**：多個 USB 3.0 感測器若插在同一個 USB 主控晶片（Host Controller）上，會因共享頻寬而導致掉幀。解決方案是將深度相機與 USB 光達分散插在不同的物理 USB 匯流排（Bus）上。
    *   **網卡設定**：對於乙太網接口的光達，建議開啟網卡的 **Ring Buffer**（接收暫存區）並啟用 **MTU 9000 (Jumbo Frames)**，大幅減輕網路中斷對 CPU 造成的負擔，徹底解決網路丟包導致的點雲掉幀。
2.  **調優 DDS 的服務品質 (QoS Policies)**：
    *   對於大頻寬的感測器話題（影像、點雲），一律將 Reliability 設為 **`Best Effort`**。這樣一來，若發生短暫的丟包，系統不會進行重傳，能保障最新的數據以最低延遲（無阻塞）送達演算法。
    *   將 History 的 `Depth`（佇列深度）設為極小值（例如 `1` 或 `2`）。這能確保當演算法處理不贏時，記憶體中不會積壓舊資料，演算法每次拿到的必定是最新採樣的一幀。
3.  **驅動多執行緒（Multi-threading）與生命週期管理**：
    *   在驅動代碼中，將「硬體資料讀取（I/O Thread）」與「ROS 2 話題發布（Publish Thread）」拆分到不同的獨立執行緒中，利用線緒安全的無鎖佇列（Lock-free Queue）傳遞資料，避免 ROS 話題發布阻塞導致硬體讀取中斷。

### 1.2 移動系統的精準校正實務

除了單一感測器的訊號流優化外，整機系統在移動時，必須對底盤、里程計以及多感測器聯合外參進行精確標定與校正。

#### 1.2.1 里程計與輪徑/軸距校正 (Odometry Calibration)
機器人的輪子在實體製造時可能存在微小的尺寸誤差，或因長期磨損導致輪徑變小；此外，左右輪的軸距誤差會直接導致機器人「走不直」或「轉向角度不準」。
*   **標定方法**：
    *   **直線測試**：命令機器人向前精準行駛 10 公尺，利用高精度雷射測距儀測量實際物理距離，計算出**輪徑修正係數**。
    *   **原地旋轉測試**：命令機器人原地精準旋轉 360 度 10 次，比對編碼器推算的角度與實際物理旋轉角度，計算出**輪距（Track Width）修正係數**。
    *   **IMU 融合**：在軟體層級，利用擴展卡爾曼濾波（EKF，如 `robot_localization` 套件），將輪子里程計與九軸 IMU 的偏航角（Yaw）速度進行緊密融合，大幅抑制輪子打滑帶來的累計位置漂移。

#### 1.2.2 相機與 IMU 聯合外參標定 (Camera-IMU Calibration)
在視覺慣性導航（VIO）中，相機與 IMU 之間的外參精度必須達到公釐（mm）級與毫弧度（mrad）級。
*   **實務工具 ── Kalibr**：
    *   **原理**：使用一個已知尺寸的標定板（如 Aprilgrid 或棋盤格），將相機與 IMU 牢固綁定，在標定板前進行多維度的翻轉與平移運動。
    *   **輸出**：Kalibr 演算法會同時估算出相機的內參（鏡頭畸變參數）、IMU 的白噪聲與隨機游走噪聲（Random Walk），以及兩者之間極其精確的時空外參。
